# ctx.reply() inside conversation.external() breaks Grammy v2 replay → waitForCallbackQuery hang

**Date:** 2026-05-01 12:00
**Files:** src/bot/conversations/settings.ts
**Severity:** high

## Problem

After pressing "SoloLeveling (синхронизировать)" in /settings, the bot froze: "Готово" button
stopped working, text commands stopped responding. Only redeploy restored the bot.
The same hang could occur for timezone (intermittent).

## Root Cause

`ctx.reply()` was called **inside** `conversation.external()` closures in the SoloLeveling
auto-sync branch (3 separate reply calls, covering success/error/401 paths).

In Grammy conversations v2, `conversation.external()` is executed once on first run and its
result cached in session. On replay (re-execution of the conversation function on every new
incoming update), the closure is **not re-executed** — only the cached result is returned.
This means any `ctx.reply()` inside `external()` fires on first run but is **skipped during
replay**. Grammy v2 tracks outgoing Telegram API calls as conversation steps; when the step
count during replay doesn't match the original run, `waitForCallbackQuery` cannot correctly
advance and the conversation hangs permanently.

Additionally, the no-token branch used `.then()` chaining on `conversation.external()` for
error handling — a non-standard pattern that bypassed proper Grammy tracking.

## Solution

1. **Auto-sync branch (has token):** Replaced all `ctx.reply()` inside `external()` with a
   discriminated return value (`'ok' | 'expired' | 'error'`). Grammy returns the cached
   string on replay; `ctx.reply()` is called OUTSIDE `external()` based on the result.

2. **No-token branch:** Replaced `.then()` error handling with try/catch inside `external()`,
   returning `{ ok: true; updatedUser }` or `{ ok: false; code }`. All `ctx.reply()` calls
   moved outside `external()`.

3. **Expired token refresh:** Changed from `.then()` on `getUserByTelegramId` to an explicit
   `async () => {}` closure with `conversation.external<DbUser>()` generic annotation to
   preserve TypeScript control flow narrowing on the `user` variable.

## Prevention

- **Never call `ctx.*` (or any Grammy API methods) inside `conversation.external()`.**
  `external()` is a black box for Grammy replay — only pure DB/HTTP/crypto operations belong
  inside it. Return values or result objects to signal outcomes.
- Pattern: `const result = await conversation.external<ResultType>(async () => { try { ... return { ok: true, data } } catch { return { ok: false } } })` then react outside.
- When assignments like `user = await conversation.external(...)` use TypeScript generics
  (`external<DbUser>()`), flow narrowing is preserved in subsequent code.
