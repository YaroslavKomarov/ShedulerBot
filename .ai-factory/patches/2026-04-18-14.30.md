# Single-round tool calling prevented sequential LLM operations (add_task never executed)

**Date:** 2026-04-18 14:30
**Files:** src/llm/agent-query.ts
**Severity:** critical

## Problem

Period start/preview notifications showed "–ó–∞–¥–∞—á–∏ –Ω–µ –∑–∞–ø–ª–∞–Ω–∏—Ä–æ–≤–∞–Ω—ã" even though the
user had added tasks via the agent. The agent confirmed "–∑–∞–¥–∞—á–∞ –¥–æ–±–∞–≤–ª–µ–Ω–∞" but tasks
were never saved to the DB ‚Äî or were saved with a wrong period_slug.

Reproduced consistently: tasks added on 17.04 and 18.04 never appeared in any
period notification (–§–∏–∑–∏—á–µ—Å–∫–∞—è –∞–∫—Ç–∏–≤–Ω–æ—Å—Ç—å, –õ–∏—á–Ω—ã–µ –ø—Ä–æ–µ–∫—Ç—ã, –£—Ç—Ä–µ–Ω–Ω—è—è —Ä—É—Ç–∏–Ω–∞,
–ü–µ—Ä–∏–æ–¥ –º–µ–Ω—Ç–æ—Ä—Å—Ç–≤–∞). Morning plan also showed all periods as empty.

## Root Cause

`handleAgentMessage` supported only **one round** of tool calls:
1. LLM first call (with `tools`) ‚Üí requests tool(s)
2. Tools executed
3. LLM second call (**without** `tools`) ‚Üí generates text response

The system prompt instructs the LLM: "—Å–Ω–∞—á–∞–ª–∞ –≤—ã–∑–æ–≤–∏ `get_periods`, –ø–æ—Ç–æ–º `add_task`".
This requires two sequential rounds. With the old architecture:

**Scenario A:** LLM follows the instruction and calls ONLY `get_periods` in round 1.
Gets the periods list. In round 2 there are no tools available ‚Üí cannot call `add_task`
‚Üí task is **never created** ‚Üí LLM hallucinates "–∑–∞–¥–∞—á–∞ –¥–æ–±–∞–≤–ª–µ–Ω–∞".

**Scenario B:** LLM calls `get_periods` + `add_task` in parallel (before seeing the
periods list), guessing the slug. Slug validation (added in previous patch) rejects the
wrong slug ‚Üí error returned ‚Üí task not created ‚Üí LLM hallucinates success in round 2
(which also had no `tools`).

The previous fix (slug validation in `add_task`) was correct but incomplete ‚Äî it caught
invalid slugs but gave the LLM no way to retry with the correct slug.

## Solution

Replaced the two-call structure with an **agentic loop** (max 5 iterations):

```typescript
while (iteration < MAX_ITERATIONS) {
  const response = await llmClient.chat.completions.create({
    model: STRONG_MODEL,
    messages,
    tools: TOOLS,           // tools available in EVERY iteration
    tool_choice: 'auto',
    temperature: 0.3,
  })
  const message = response.choices[0].message
  messages.push(message)

  if (!message.tool_calls || message.tool_calls.length === 0) {
    return message.content  // LLM decided it's done
  }

  // execute tools, push results, loop continues
}
```

Now the LLM can:
1. Iteration 1: call `get_periods` ‚Üí receives real slugs
2. Iteration 2: call `add_task` with correct slug ‚Üí task created
3. Iteration 3: generate final response (no tool_calls) ‚Üí loop ends

Added `[FIX]` logging: iteration count, tools called per round, final token usage.

## Prevention

- Any LLM agent that needs sequential tool operations (lookup ‚Üí mutate) requires
  a multi-round loop, not a two-shot call. Always design with an agentic loop.
- The two-shot pattern (first call with tools, second without) is only safe when
  ALL needed data is already in the context and NO lookup is needed before mutation.
- When the system prompt says "first call X then call Y", verify the architecture
  supports multiple rounds ‚Äî otherwise the instruction is impossible to follow.
- Add a test that verifies `handleAgentMessage` with a mock that requires 2 rounds:
  first `get_periods`, then `add_task`. The task should appear in DB after the call.
