# Tasks scheduled for today without period_slug are invisible to notifications

**Date:** 2026-04-15 12:00
**Files:** src/db/tasks.ts, src/cron/morning-plan.ts, src/cron/period-notify.ts, src/lib/date.ts
**Severity:** high

## Problem

Tasks created via the agent with `scheduled_date = today` but `period_slug = null` never
appeared in morning-plan notifications or period-start notifications. They were only visible
when queried directly via the agent.

Additionally, `getTodayInTimezone` used JS day-of-week encoding (0=Sun…6=Sat) while all
period data was stored with ISO encoding (1=Mon…7=Sun), causing Sunday periods to be
invisible in all queries.

## Root Cause

**Bug 1 — unassigned tasks**: `getTaskQueue(userId, periodSlug, date)` always filters by
`period_slug = periodSlug`. Tasks with `period_slug = null` are skipped. `getBacklog` also
skips them because it requires `scheduled_date IS NULL`. A task with `scheduled_date = today`
and `period_slug = null` matched neither query — invisible to all notifications, but visible
to the agent's `getTasksByDate` (which has no period filter).

Root cause of null period_slug: the LLM sometimes omits `period_slug` when calling `add_task`
even though the system prompt instructs it to ask. Also: retrospective "На завтра" only updates
`scheduled_date`, so if the original task lacked a period, it stays without one.

**Bug 2 — Sunday periods**: `getTodayInTimezone` produced `dayOfWeek = 0` for Sunday (JS
`new Date().getDay()` convention), but DB stores ISO Sunday as `7`. Sunday periods were
never found by `getPeriodsForDay(userId, 0)`.

**Bug 3 — timezone**: Agent computed `today` as `new Date().toISOString().split('T')[0]`
(UTC), while `morning-plan` used user's timezone. Dates could mismatch for users in negative
UTC offsets around midnight.

## Solution

1. Extracted `getTodayInTimezone` to `src/lib/date.ts` with ISO encoding (1=Mon…7=Sun).
   Re-exported from `morning-plan.ts` for backward compatibility.

2. Fixed `agent-query.ts` to use `getTodayInTimezone(user.timezone)` instead of UTC date.

3. Added `getUnassignedTodayTasks(userId, date)` in `tasks.ts`: returns tasks with
   `scheduled_date = date AND period_slug IS NULL`.

4. In `morning-plan.ts`: after building period plans, calls `getUnassignedTodayTasks` and
   appends results to the first period's task list (deduped by task id).

5. In `period-notify.ts` `sendPeriodPreview` and `sendPeriodStart`: also calls
   `getUnassignedTodayTasks` and merges results into the task list shown.

## Prevention

- When the LLM creates tasks, consider making `period_slug` required at the tool level
  (add to `required` array in the tool definition). This forces the LLM to either provide
  it or ask the user explicitly.
- When displaying "task is in a period" to users, show the actual `period_slug` so
  discrepancies are visible.
- Add a test that verifies tasks with `period_slug = null, scheduled_date = today` appear
  in morning-plan output.
