# Period notifications showed entire backlog instead of slotted tasks

**Date:** 2026-04-27 12:00
**Files:** src/cron/period-notify.ts
**Severity:** high

## Problem

`sendPeriodEnd` and `sendPeriodPreview` displayed ALL pending tasks in a period
(e.g. 60 backlog tasks), not just the 3–4 tasks that were supposed to fit into the
period's time window that day. `sendPeriodPreview` also showed wrong overflow counts
("...и ещё 55" instead of 0).

## Root Cause

Both functions called `getTaskQueue(userId, periodSlug, date)` directly. This query
returns ALL pending tasks in a period where `scheduled_date = date OR scheduled_date
IS NULL` — i.e. the entire backlog of that period. No time-budget filtering was applied.

`sendPeriodStart` already had the correct pattern: `buildSlots` + `injectGuaranteedTasks`
to limit display to only tasks that fit within the period's time window. The other two
notifications were written without this logic, likely before `buildSlots` existed.

## Solution

Applied the same `buildSlots` + `injectGuaranteedTasks` pattern to both functions.
All three period notification functions (preview, start, end) now consistently show
only the tasks allocated to the period's time budget.

Added `[FIX]` logging: `totalQueue` vs `slotted`/`plannedPending` counts for verification.

## Prevention

- When adding a new notification that shows tasks for a period, always use `buildSlots`
  to filter by time budget. Direct `getTaskQueue` is only appropriate when you intentionally
  want the full unfiltered queue (e.g. for capacity checks).
- The pattern is: `getTaskQueue` → `buildSlots` → `injectGuaranteedTasks` → display.
  Any deviation requires a conscious comment explaining why.
