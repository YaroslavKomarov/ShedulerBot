## Шаг 1 — Фундамент ✅

> Миграция применялась вручную через Supabase Dashboard. Начиная с Шага 2 — использовать Supabase CLI (`supabase db push`).

- [x] Инициализировать `package.json`, установить зависимости:
  - `grammy`, `@grammyjs/conversations`, `@grammyjs/types`
  - `@supabase/supabase-js`
  - `openai` (OpenRouter-совместимый клиент)
  - `googleapis`
  - `node-cron`
  - `express`
  - `zod`
  - `typescript`, `tsx`, `@types/node`, `@types/express`
- [x] Настроить `tsconfig.json` (target ES2022, module NodeNext)
- [x] Создать `.env.example` со всеми переменными окружения
- [x] Создать схему БД — SQL-миграция для таблиц `sch_users`, `sch_periods`, `sch_tasks` (с индексами)
- [x] Применить миграцию в Supabase (вручную через Dashboard)
- [x] Сгенерировать TypeScript-типы из схемы (`npx supabase gen types typescript`)
- [x] Реализовать `src/db/client.ts` — Supabase клиент с service role key
- [x] Реализовать базовые query-функции: `getUserByTelegramId`, `createUser`
- [x] Реализовать `src/llm/client.ts` — OpenRouter клиент + константы моделей
- [x] Реализовать `src/bot/index.ts` — инициализация Grammy бота, session middleware, conversations middleware
- [x] Реализовать `src/index.ts` — точка входа: Express + bot (polling в dev, webhook в prod)
- [x] Команда `/start` отвечает "Привет!" (заглушка до онбординга)
- [x] Убедиться что бот запускается локально (`npm run dev`)

## Шаг 2 — Онбординг ✅

**Цель:** новый пользователь проходит диалог, в результате создаются профиль и периоды в БД, регистрируются cron-джобы.

- [x] Реализовать `src/db/users.ts` — полный CRUD пользователя (`createUser`, `updateUser`, `getAllUsersForCron`)
- [x] Реализовать `src/db/periods.ts` — создание и чтение периодов (`createPeriods`, `getPeriodsForDay`, `getUserPeriods`)
- [x] Реализовать `src/llm/interview.ts` — `continueInterview()` + системный промпт онбординга
- [x] Реализовать парсинг результата интервью: извлечь JSON из `<data>...</data>` тегов
- [x] Реализовать `src/bot/conversations/onboarding.ts`:
  - Диалог до получения таймзоны, периодов, времени утра/конца дня
  - Подтверждение итогов пользователем
  - Опциональный Google Calendar OAuth (отправка ссылки)
  - Сохранение профиля и периодов в БД
- [x] Реализовать `src/cron/manager.ts`:
  - `registerUserCrons(user)` — регистрирует все джобы для одного пользователя
  - `unregisterUserCrons(userId)` — удаляет джобы
  - `registerAllUsers()` — при старте сервиса регистрирует джобы всех пользователей
- [x] Вызов `registerUserCrons` в конце онбординга
- [x] Вызов `registerAllUsers` при старте сервиса
- [x] Команда `/start`: если пользователь новый → войти в `onboarding` conversation; если существующий → показать план на сегодня (заглушка)

## Шаг 3 — Добавление и управление задачами ✅

**Цель:** пользователь может добавлять задачи через свободный текст; задачи хранятся в правильном порядке приоритетов.

- [x] Реализовать `src/db/tasks.ts`:
  - `createTask` — создание задачи
  - `updateTask` — обновление полей
  - `getTaskQueue(userId, periodSlug, date)` — очередь с сортировкой: urgent → deadline → created_at
  - `getBacklog(userId, periodSlug?)` — задачи без `scheduled_date`
  - `getTasksByDate(userId, date)` — все задачи на дату
  - `findTasksByReference(userId, reference)` — поиск по семантическому совпадению заголовка
- [x] Реализовать `src/llm/parse-task.ts` — `parseTaskMessage()`: извлечение структурированной задачи из свободного текста (fast model)
- [x] Реализовать `src/llm/intent.ts` — `detectIntent()`: классификация намерений пользователя (fast model)
- [x] Реализовать `src/bot/conversations/add-task.ts`:
  - Парсинг сообщения → черновик задачи
  - Уточнение если формулировка размытая
  - Генерация описания если отсутствует (предложить подтвердить)
  - Предложение оценки времени если не указана
  - Запрос даты (`сегодня` / `завтра` / конкретная / без даты)
  - Сохранение задачи в БД + подтверждение пользователю
- [x] Реализовать `src/bot/handlers.ts` — catch-all обработчик свободного текста:
  - Определяет `intent` через LLM
  - Маршрутизирует: `add_task` → conversation, остальные → inline handlers
- [x] Обработчик `modify_task`: изменение срочности, даты, отмена задачи
- [x] Обработчик `show_plan` / `show_backlog` — заглушки (реализуются в шаге 4/5)

## Шаг 4 — Утренний план и уведомления периодов ✅

**Цель:** утром приходит план, перед/после каждого периода — уведомления; план синкается в Google Calendar.

- [x] Реализовать `src/llm/plan.ts` — `generateDayPlanMessage()`: форматирование плана (strong model)
- [x] Реализовать `src/calendar/auth.ts` — `getAuthUrl()`, `createOAuthClient()`
- [x] Реализовать `src/calendar/client.ts` — `getCalendarClient(userId)` с авто-рефрешем токенов
- [x] Реализовать `src/calendar/sync.ts` — `syncDayPlan()`: каждый период → событие с задачами в описании
- [x] Реализовать `src/routes/auth.ts` — GET `/auth/google/callback`: сохранение токенов после OAuth
- [x] Реализовать `src/cron/morning-plan.ts`:
  - Определить активные периоды на сегодня (по `days_of_week`)
  - Для каждого периода — задачи из очереди (приоритет + бэклог если места остаётся)
  - Рассчитать тайм-слоты задач на основе `estimated_minutes`
  - Отправить план пользователю
  - Синкнуть в Google Calendar (если подключён)
- [x] Реализовать `src/cron/period-notify.ts` — 4 уведомления на каждый период:
  - За 10 минут до начала — превью задач
  - В момент начала — список задач
  - За 10 минут до конца — напоминание отметить выполненное
  - В момент конца — запрос итогов + переход к следующему периоду
- [x] Зарегистрировать в `cron/manager.ts`:
  - Джоб утреннего плана (по `morning_time` пользователя)
  - 4 джоба для каждого активного периода
- [x] Команда `/plan` — отправить план на сегодня по запросу
- [x] Команда `/tomorrow` — отправить план на завтра

## Шаг 5 — Управление задачами внутри периода ✅

**Цель:** пользователь может отмечать выполненные, обновлять прогресс, видеть следующие из очереди.

- [x] Обработчик `mark_done` в `handlers.ts`:
  - LLM определяет какие задачи выполнены из сообщения пользователя
  - Обновить `status = done` для найденных задач
  - Ответить: список выполненных + остаток в периоде
- [x] Обработчик `update_progress` в `handlers.ts`:
  - LLM обновляет `title`, `description`, `progress_note`, `estimated_minutes`
  - Статус остаётся `pending`
  - Ответить: обновлённая формулировка задачи
- [x] Логика "все задачи выполнены, время ещё есть":
  - Показать 5 следующих задач из очереди периода
  - Пользователь выбирает любую
- [x] Команда `/backlog` — показать задачи без даты по периодам

## Шаг 6 — Ретроспектива конца дня ✅

**Цель:** вечером бот подводит итоги, помогает разобраться с невыполненным и задачами без срока.

- [x] Реализовать `src/cron/retrospective.ts`:
  - Собрать выполненные и невыполненные задачи за день
  - Отправить сводку (этап 1)
- [x] Реализовать `src/bot/conversations/retrospective.ts`:
  - **Этап 1** — невыполненные задачи: пользователь распределяет по датам / меняет срочность / пропускает (→ переносятся на завтра)
  - **Этап 2** — задачи без даты и срочности: назначить срок, сделать срочными, или пропустить
  - Завершение: "Спокойной ночи!"
- [x] Поведение по умолчанию при `пропусти`:
  - Этап 1: невыполненные → `scheduled_date = tomorrow`
  - Этап 2: без изменений
- [x] Зарегистрировать джоб ретроспективы в `cron/manager.ts` (по `end_of_day_time`)

## Шаг 7 — Внешний API ✅

**Цель:** другие модули системы (SoloLeveling и др.) могут добавлять задачи в ShedulerBot через HTTP.

- [x] Реализовать `src/api/tasks.ts` — `POST /api/tasks`:
  - Валидация тела запроса через Zod
  - Поля: `telegram_id`, `title`, `description?`, `period_slug`, `deadline_date?`, `estimated_minutes?`, `is_urgent`, `source`, `external_id?`
  - Найти пользователя по `telegram_id` → если не найден, вернуть 404
  - Создать задачу в БД
  - Отправить уведомление пользователю в Telegram
  - Вернуть `{ id, created: true }`
- [x] Добавить простую API-аутентификацию (shared secret в заголовке `X-Api-Key`)
- [x] Обработка дублей: если `external_id` уже существует → вернуть существующую задачу (idempotent)
- [x] Подключить router в `src/index.ts`

**Цель:** пользователь может отправлять голосовые сообщения вместо текста — бот транскрибирует и обрабатывает их как обычный текст.

- [x] Реализовать `src/bot/middleware/voice.ts` — скачать `.ogg` файл через Bot API, отправить в Whisper (OpenRouter), вернуть транскрипцию
- [x] Подключить middleware в `src/bot/index.ts` перед catch-all обработчиком: `message:voice` → транскрибировать → передать текст в `handleText()`
- [x] Добавить константу модели Whisper в `src/llm/client.ts`
- [x] Обработка ошибок: если транскрипция не удалась — ответить "Не удалось распознать голосовое сообщение, попробуй текстом"
