# LLM invents period_slug instead of using actual DB slug

**Date:** 2026-04-17 14:10
**Files:** src/llm/agent-query.ts, src/db/tasks.ts
**Severity:** critical

## Problem

Period start/preview notifications showed "Задачи не запланированы" even though tasks
existed in DB and were findable via direct agent query. Tasks were created with correct
`scheduled_date` and `status = 'pending'`, but `getTaskQueue` returned 0 results.

## Root Cause

The LLM calls `add_task` with an invented `period_slug` (e.g. "mentorship", "work-1")
instead of looking up the actual slug from `get_periods`. The task is stored with the wrong
slug. `getTasksByDate` (no period filter) finds it; `getTaskQueue(period.slug, date)` misses
it because `period_slug` in the task doesn't match the actual period slug.

The previous fix (checking `period_slug IS NULL`) didn't help because the field was not
null — it had a wrong non-null value. There was no validation at creation time.

## Solution

In `executeTool` for `add_task`: after extracting `period_slug`, call `getUserPeriods` and
verify the slug exists. If not, return an error response listing available periods with their
correct slugs. The LLM sees this error in the tool result and retries with a correct slug
(or asks the user which period to use).

Also bumped `getTaskQueue` result log from `debug` to `info` so it's visible in production
and can confirm what's actually returned during notifications.

## Prevention

- Any tool that takes a slug/id referencing an existing DB record should validate it before
  writing. "Slug exists" check is cheap (reuses the already-loaded periods list).
- When the LLM must reference an existing entity by slug/id, the system prompt should
  explicitly say "first call get_X to get the list, then use the exact slug from that list."
  The `add_task` description now says this.
